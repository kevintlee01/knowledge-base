What Are Data Structures & Algorithms?

The foundational building blocks of computer science — how we organize data and the step-by-step procedures we use to solve computational problems efficiently.

📐

Data Structures

A data structure is a specialized format for organizing, processing, retrieving, and storing data. Different structures are suited for different kinds of applications — some are highly specialized for specific tasks, while others are general-purpose.

The choice of data structure often determines the efficiency of a program. A hash map provides O(1) average-case lookups, while a sorted array enables O(log n) binary search. Understanding trade-offs between structures is critical for building performant software.

⚡

Algorithms

An algorithm is a finite sequence of well-defined instructions to solve a class of problems or perform a computation. Algorithms are independent of programming languages — they are abstract procedures that can be implemented in any language.

Algorithm design involves choosing the right strategy (divide and conquer, greedy, dynamic programming, backtracking) and analyzing correctness, time complexity, and space complexity. The ultimate goal is finding the most efficient solution within given constraints.

Complexity Analysis

Big O notation and asymptotic analysis — the universal language for describing algorithm efficiency and scalability.

📏

Big O Notation

Big O describes the upper bound of time or space growth as input size approaches infinity. It abstracts away constants and lower-order terms, focusing on the dominant factor.

  • O(1): Constant — hash map lookup
  • O(log n): Logarithmic — binary search
  • O(n): Linear — array traversal
  • O(n log n): Linearithmic — merge sort
  • O(n²): Quadratic — bubble sort
  • O(2ⁿ): Exponential — subset generation
⏱️

Time Complexity

Measures the number of elementary operations an algorithm performs as a function of input size. We analyze three cases:

  • Best Case (Ω): Minimum operations needed
  • Average Case (Θ): Expected operations over all inputs
  • Worst Case (O): Maximum operations — most commonly used
  • Amortized: Average over a sequence of operations (e.g., dynamic array resize)
💾

Space Complexity

Measures the total memory an algorithm uses relative to input size, including auxiliary space (extra memory beyond the input).

  • In-place: O(1) extra space (e.g., insertion sort)
  • Linear space: O(n) extra (e.g., merge sort temp array)
  • Stack space: Recursive calls add O(depth) space
  • Trade-off: Often you trade space for time (memoization, caching)

Python Built-in Data Types

From simple integers and strings to powerful dictionaries and sets — Python's built-in types are the building blocks for every data structure and algorithm. Master their operations and Big O characteristics.

🔢 Integers & Floats — Numeric Types

Python integers (int) have arbitrary precision — they can be as large as your memory allows. Floats (float) are 64-bit IEEE 754 double-precision numbers. All arithmetic operations are O(1) for fixed-size integers.

Big O: Arithmetic (+, -, *, //, %) is O(1) for bounded integers. For arbitrarily large integers of n digits, addition is O(n) and multiplication is O(n log n).
Space: int uses 28 bytes (small values) up to proportional to digit count. float is always 24 bytes.

✅ Booleans — True/False Logic

bool is a subclass of int in Python — True is 1, False is 0. All comparison and logical operations are O(1). Understanding truthiness is critical for writing clean Python.

Big O: All boolean operations (and, or, not, comparisons) are O(1).
Falsy values: False, 0, 0.0, "", [], (), {}, set(), None

📝 Strings — Immutable Text Sequences

Python strings (str) are immutable sequences of Unicode characters. Once created, they cannot be modified — every operation returns a new string. This immutability is key to understanding why repeated string concatenation is O(n²).

Big O for strings of length n:
Access s[i] → O(1)  |  Search (in) → O(n)  |  Slice s[a:b] → O(b-a) — creates a copy
Concatenation + → O(n+m) — creates new string  |  s.join() → O(total) — optimal for building strings
s.find() / s.index() → O(n·m) worst case  |  len(s) → O(1) — stored as attribute
Space: O(n) — 49 bytes + 1 byte per char (ASCII), up to 4 bytes per char (Unicode)

📦 Lists — Dynamic Arrays

Python list is a dynamic array — a contiguous, resizable block of references to objects. It is the most used data structure in Python and supports heterogeneous elements. Internally, it over-allocates space and doubles capacity when full (amortized O(1) append).

Big O for list of length n:
Access list[i] → O(1)  |  Search (in) → O(n) — linear scan  |  Append → O(1) amortized
Insert at i → O(n) — shifts elements  |  Delete at i → O(n) — shifts elements  |  Pop last → O(1)
Pop at i → O(n)  |  Slice [a:b] → O(b-a) — copies  |  Sort → O(n log n) — Timsort
Extend → O(k) where k = len of iterable  |  len() → O(1)  |  min/max → O(n)

📎 Tuples — Immutable Sequences

Tuples (tuple) are immutable ordered sequences. They are faster than lists, use less memory, and can be used as dictionary keys or set elements (because they're hashable, if all their elements are hashable). Same Big O as lists for read operations.

Big O for tuple of length n:
Access t[i] → O(1)  |  Search (in) → O(n)  |  Slice t[a:b] → O(b-a)
len(t) → O(1)  |  Concatenation + → O(n+m) — creates new tuple
No mutation: No append, insert, remove, pop. This is by design.
Memory: Tuples use ~56 bytes + 8 bytes per element (vs ~56 + 8 for list with extra preallocation overhead)

📖 Dictionaries — Hash Maps in Python

Python dict is a hash table — the most important data structure for coding interviews. It maps hashable keys to values with average O(1) lookups, inserts, and deletes. Since Python 3.7, dicts maintain insertion order.

Big O for dict with n key-value pairs:
Get d[key] → O(1) avg, O(n) worst  |  Set d[key] = val → O(1) avg
Delete del d[key] → O(1) avg  |  Search (key in d) → O(1) avg — this is the key advantage!
Iterate keys/values → O(n)  |  len(d) → O(1)
Worst case: O(n) occurs with many hash collisions (extremely rare with Python's hash function).
Space: O(n) — each entry stores hash, key pointer, and value pointer. Empty dict is 64 bytes.

🎯 Sets — Unique Collections with O(1) Lookup

Python set is an unordered collection of unique hashable elements, backed by a hash table. It provides O(1) average membership testing — drastically faster than checking x in list (which is O(n)). Also supports mathematical set operations.

Big O for set with n elements:
Add s.add(x) → O(1) avg  |  Remove s.remove(x) → O(1) avg  |  Search (x in s) → O(1) avg
Union s | t → O(n+m)  |  Intersection s & t → O(min(n,m))  |  Difference s - t → O(n)
Subset s <= t → O(n)  |  len(s) → O(1)  |  Pop s.pop() → O(1)
Key rule: Elements must be hashable (immutable): int, float, str, tuple. Lists and dicts can NOT be set elements.
frozenset: Immutable version of set — can be used as dict key or nested in another set.

Data Structures Overview

A taxonomy of the fundamental data structures — from simple linear structures to complex graph-based organizations.

📦 Arrays

Contiguous memory blocks with O(1) index access. Fixed-size (static) or resizable (dynamic/ArrayList). Foundation for many algorithms.

  • Access: O(1) by index
  • Search: O(n) unsorted, O(log n) sorted
  • Insert/Delete: O(n) — requires shifting elements
  • Best for: Random access, iteration, cache-friendly ops

🔗 Linked Lists

Nodes connected via pointers. Singly, doubly, or circular variants. Dynamic size with efficient insertions at any position.

  • Access: O(n) — must traverse from head
  • Insert/Delete (at known pos): O(1)
  • Search: O(n) — linear scan required
  • Best for: Frequent insertions/deletions, implementing queues

📚 Stacks (LIFO)

Last-In-First-Out discipline. Push/pop from top only. Used in parsing, undo systems, DFS, call stacks, expression evaluation.

  • Push/Pop/Peek: O(1)
  • Applications: Balanced parentheses, postfix evaluation, backtracking

🚦 Queues (FIFO)

First-In-First-Out discipline. Enqueue at rear, dequeue from front. Used in BFS, scheduling, buffering, and message queues.

  • Enqueue/Dequeue: O(1)
  • Variants: Circular queue, deque (double-ended), priority queue

🌳 Trees

Hierarchical data with a root node and children. Fundamental for representing hierarchical relationships, search, and sorting.

  • Binary Tree: At most 2 children per node
  • BST: Left < root < right — O(log n) avg search
  • AVL/Red-Black: Self-balancing — guaranteed O(log n)
  • B-Tree: Disk-optimized, used in databases and filesystems

📊 Heaps

Complete binary tree satisfying the heap property. Min-heap: parent ≤ children. Max-heap: parent ≥ children.

  • Insert: O(log n) — bubble up
  • Extract-Min/Max: O(log n) — bubble down
  • Build heap: O(n) — bottom-up construction
  • Used in: Priority queues, heap sort, Dijkstra's

🕸️ Graphs

Vertices connected by edges. Directed or undirected, weighted or unweighted, cyclic or acyclic. Model networks, maps, dependencies.

  • Adjacency Matrix: O(V²) space, O(1) edge lookup
  • Adjacency List: O(V+E) space, efficient iteration
  • Traversals: BFS (level-order), DFS (depth-first)
  • Key algorithms: Dijkstra, Bellman-Ford, Kruskal, Prim

🔤 Tries

Prefix tree for efficient string operations. Each node represents a character, paths from root form stored strings.

  • Insert/Search/Delete: O(m) where m = key length
  • Prefix search: O(m) — find all words with given prefix
  • Used in: Autocomplete, spell checkers, IP routing

#️⃣ Hash Tables / Hash Maps

Key-value pairs using a hash function to compute an index. The workhorse of constant-time lookups.

  • Insert/Search/Delete: O(1) average, O(n) worst case
  • Collision handling: Chaining (linked list at each bucket) or Open addressing (linear/quadratic probing)
  • Load factor: n/m ratio — resize when too high (typically > 0.75)
  • Used in: Caches, symbol tables, counting, deduplication

🎯 Hash Sets

Collection of unique elements with O(1) membership testing. Backed by a hash table without values.

  • Add/Remove/Contains: O(1) average
  • Set operations: Union, intersection, difference — all O(n)
  • Used in: Deduplication, fast membership checks, two-sum problems

🌲 Segment Trees

Divide-and-conquer tree for range queries (sum, min, max) and point updates in O(log n) time. Built on array representation.

  • Build: O(n)
  • Query/Update: O(log n)
  • Lazy propagation: Range updates in O(log n)

📊 Binary Indexed Tree (Fenwick)

Two Pointer Technique

Use two pointers moving towards each other (or in the same direction) to solve problems in O(n) time instead of O(n²). Works on sorted arrays or when you need to find pairs/subarrays satisfying a condition.

  • Pattern: Initialize left=0, right=n-1. Move inward based on condition.
  • Applications: Two Sum (sorted), 3Sum, Container With Most Water, Remove Duplicates, Palindrome check
  • Key insight: Eliminates the need for nested loops by making intelligent pointer movement decisions

def two_sum_sorted(nums, target): left, right = 0, len(nums) - 1 while left < right: curr_sum = nums[left] + nums[right] if curr_sum == target: return [left, right] elif curr_sum < target: left += 1 else: right -= 1 return []
Sliding Window

Maintain a window [left, right] that slides across the array. Expand right to include elements, shrink left when the window violates constraints. Converts O(n²) brute-force subarray scans to O(n).

  • Fixed size: Window of size k — move both pointers together
  • Variable size: Expand right, shrink left when invalid — track best valid window
  • Applications: Longest substring without repeating chars, minimum window substring, max sum subarray of size k

Prefix Sum

Precompute cumulative sums so any subarray sum can be calculated in O(1). Build a prefix array where prefix[i] = sum of elements from index 0 to i-1.

  • Applications: Range sum queries, subarray sum equals k (with hash map), equilibrium index

Singly & Doubly Linked List — Node Structure & Building Blocks

A linked list is a sequence of nodes where each node stores a value and a reference (pointer) to the next node. Unlike arrays, elements are not stored contiguously in memory — each node can be anywhere in the heap.

Singly Linked: Each node has val and next — traverse forward only
Doubly Linked: Each node has val, next, and prev — traverse both directions
Access: O(n) — must traverse from head   Insert/Delete at known position: O(1)
Search: O(n) — no random access, no binary search

Fast & Slow Pointers (Floyd's Algorithm)

The two-pointer technique for linked lists: slow moves 1 step, fast moves 2 steps. When fast reaches the end, slow is at the middle. If there's a cycle, they will meet.

# Find middle node — O(n) time, O(1) space def find_middle(head): slow = fast = head while fast and fast.next: slow = slow.next fast = fast.next.next return slow # middle (or 2nd middle if even) # Detect cycle — Floyd's Tortoise & Hare def has_cycle(head): slow = fast = head while fast and fast.next: slow = slow.next fast = fast.next.next if slow == fast: return True return False # Find cycle start node def detect_cycle_start(head): slow = fast = head while fast and fast.next: slow = slow.next fast = fast.next.next if slow == fast: # Reset one pointer to head slow = head while slow != fast: slow = slow.next fast = fast.next return slow # cycle entry point return None # Check if palindrome — O(n) time, O(1) space def is_palindrome(head): # Find middle slow = fast = head while fast and fast.next: slow = slow.next fast = fast.next.next # Reverse second half prev = None while slow: nxt = slow.next slow.next = prev prev = slow slow = nxt # Compare both halves left, right = head, prev while right: if left.val != right.val: return False left = left.next right = right.next return True
Why Floyd's works (cycle start): When slow and fast meet inside the cycle, the distance from head to cycle start equals the distance from meeting point to cycle start (going around). This is a classic proof based on modular arithmetic.

Binary Tree — Node Structure & Fundamentals

A binary tree is a hierarchical structure where each node has at most two children (left and right). Understanding tree terminology is essential before tackling traversals and BSTs.

Root: Top node (no parent)   Leaf: Node with no children   Height: Longest path from root to leaf
Depth: Distance from root to a node   Full: Every node has 0 or 2 children   Complete: All levels filled except possibly last (filled left to right)
Perfect: All internal nodes have 2 children and all leaves at same level   Balanced: Height ≈ O(log n)

Self-Balancing Trees — AVL, Red-Black, B-Trees

Unbalanced BSTs degrade to O(n) operations. Self-balancing trees guarantee O(log n) by restructuring after inserts/deletes. You rarely implement these from scratch but must understand the concepts.

AVL Trees:
• Balance factor: height(left) - height(right) ∈ {-1, 0, 1}
• Four rotation cases after imbalance:
  - LL (Right Rotate): Insert into left child's left subtree
  - RR (Left Rotate): Insert into right child's right subtree
  - LR (Left-Right): Left rotate child, then right rotate node
  - RL (Right-Left): Right rotate child, then left rotate node
• Stricter balance than Red-Black → faster lookups, slower inserts
Red-Black Trees:
• 5 properties: (1) Every node is red or black (2) Root is black (3) Leaves (NIL) are black (4) Red node's children are both black (5) All paths from a node to descendants have same black-height
• Guarantees height ≤ 2·log₂(n+1)
• Used in: Python's sortedcontainers, Java TreeMap, C++ std::map, Linux kernel CFS scheduler
B-Trees / B+ Trees:
• Multi-way search tree: each node holds up to M keys and M+1 children
• Minimizes disk I/O — nodes sized to match disk pages
• Used in: Database indexes (PostgreSQL, MySQL InnoDB), filesystems (NTFS, ext4, HFS+)
• B+ tree: all data in leaves, internal nodes only have keys → better range scans

BFS & DFS — Full Implementations

BFS explores level by level using a queue — guarantees shortest path in unweighted graphs. DFS explores as deep as possible using recursion or a stack — natural for path-finding and cycle detection. Both run in O(V + E).

Dijkstra's Algorithm — Shortest Path (Weighted)

Dijkstra's finds the shortest path from a single source to all other vertices in a graph with non-negative edge weights. Uses a min-heap (priority queue) for O((V+E) log V) time.

Union-Find (Disjoint Set Union)

Union-Find tracks which elements belong to the same group. With path compression + union by rank, both find and union run in nearly O(1) amortized — O(α(n)) where α is the inverse Ackermann function.

class UnionFind: def __init__(self, n): self.parent = list(range(n)) self.rank = [0] * n self.components = n def find(self, x): if self.parent[x] != x: self.parent[x] = self.find(self.parent[x]) # path compression return self.parent[x] def union(self, x, y): px, py = self.find(x), self.find(y) if px == py: return False # already connected # union by rank if self.rank[px] < self.rank[py]: px, py = py, px self.parent[py] = px if self.rank[px] == self.rank[py]: self.rank[px] += 1 self.components -= 1 return True def connected(self, x, y): return self.find(x) == self.find(y) # ---- KRUSKAL'S MST using Union-Find ---- def kruskal_mst(n, edges): """edges: [(weight, u, v), ...]""" edges.sort() # sort by weight uf = UnionFind(n) mst_cost = 0 mst_edges = [] for w, u, v in edges: if uf.union(u, v): mst_cost += w mst_edges.append((u, v, w)) return mst_cost, mst_edges # ---- Connected Components ---- def count_components(n, edges): uf = UnionFind(n) for u, v in edges: uf.union(u, v) return uf.components

Recursively split the array in half, sort each half, then merge the two sorted halves. Guaranteed O(n log n) regardless of input — the go-to when stability and worst-case performance matter.

Pick a pivot, partition the array so all elements less than pivot are on the left and greater on the right, then recurse. In-place and cache-friendly — the most commonly used sort in practice.

Strategy: Choose pivot → partition around it → recurse on left and right partitions

Stable: No — partition swaps can reorder equal elements

Build the sorted array one element at a time by inserting each new element into its correct position. Adaptive and stable — runs in O(n) on nearly-sorted arrays, making it ideal as a base case for hybrid sorts like Timsort.

Time: O(n) best (sorted) / O(n²) average and worst   Space: O(1) in-place
Key property: Adaptive — fewer swaps when data is nearly sorted

Repeatedly step through the array, compare adjacent elements, and swap them if they're in the wrong order. The largest unsorted element "bubbles up" to its correct position each pass. Simple but inefficient — mainly used for educational purposes.

Strategy: Compare adjacent pairs → swap if out of order → repeat until sorted
Time: O(n) best (already sorted with early exit) / O(n²) average and worst   Space: O(1)
Stable: Yes — equal elements are never swapped past each other

Build a max-heap from the array, then repeatedly extract the maximum and place it at the end. Guaranteed O(n log n) in all cases and fully in-place — but not cache-friendly and not stable.

Searching Algorithms

From linear scan to logarithmic binary search to graph traversals — the fundamental techniques for finding what you need, fast.

🔎

Binary Search

O(log n) search on sorted arrays. The key is identifying the correct invariant — what does "left" and "right" represent at each step?

  • Standard: Find exact target in sorted array
  • Lower bound: First position where element ≥ target
  • Upper bound: First position where element > target
  • Answer space: Binary search on the answer rather than the data
🌊

Graph Traversals

Two fundamental ways to explore a graph or tree — each optimal for different problem types.

  • BFS: Level-by-level exploration using a queue — guarantees shortest unweighted path
  • DFS: Deep exploration using a stack/recursion — natural for backtracking and path finding
  • When to use BFS: Shortest path, level-order, nearest neighbor
  • When to use DFS: Cycle detection, topological sort, connected components
Linear Search — O(n) Sequential Scan

The simplest search algorithm — examine each element one by one until the target is found or the array ends. Works on unsorted data. Often overlooked, but it's the only option when data isn't sorted and preprocessing isn't worth the cost.

Time: O(n) worst and average, O(1) best   Space: O(1)
When to use: Unsorted data, small arrays, or single-use queries where sorting overhead isn't justified

def linear_search(arr, target): for i in range(len(arr)): if arr[i] == target: return i return -1 # Example print(linear_search([4, 2, 7, 1, 9], 7)) # 2 print(linear_search([4, 2, 7, 1, 9], 5)) # -1
Binary Search — O(log n) Divide-and-Eliminate

On a sorted array, compare the target with the middle element to eliminate half the search space each step. The three standard templates: exact match (left <= right), lower bound (left < right), and upper bound.

Prerequisite: Array must be sorted (or have a monotonic property)
Time: O(log n)   Space: O(1) iterative / O(log n) recursive
Key detail: Use mid = left + (right - left) // 2 to avoid integer overflow

# Standard binary search — exact match def binary_search(arr, target): left, right = 0, len(arr) - 1 while left <= right: mid = left + (right - left) // 2 if arr[mid] == target: return mid elif arr[mid] < target: left = mid + 1 else: right = mid - 1 return -1 # Lower bound — first index where arr[i] >= target def lower_bound(arr, target): left, right = 0, len(arr) while left < right: mid = (left + right) // 2 if arr[mid] < target: left = mid + 1 else: right = mid return left # Upper bound — first index where arr[i] > target def upper_bound(arr, target): left, right = 0, len(arr) while left < right: mid = (left + right) // 2 if arr[mid] <= target: left = mid + 1 else: right = mid return left # Example arr = [1, 3, 3, 5, 7, 9] print(binary_search(arr, 5)) # 3 print(lower_bound(arr, 3)) # 1 (first 3) print(upper_bound(arr, 3)) # 3 (past last 3)
Binary Search on Answer Space — Searching the Result, Not the Data

Instead of searching an array, binary search the answer itself. Define a range of possible answers [lo, hi], then for each midpoint, check if it's a feasible solution. This pattern appears in optimization problems: "find the minimum/maximum value that satisfies a constraint."

Template: Define [lo, hi] bounds → for each mid, run a feasibility check → narrow the range
Time: O(n × log(range)) — n for feasibility check, log(range) for binary search
When to use: "Minimize the maximum" or "find minimum capacity/speed that satisfies X"

Breadth-First Search (BFS) — Level-by-Level Exploration

Explore a graph or tree level by level using a queue. BFS visits all nodes at distance d before any node at distance d+1, making it optimal for finding shortest unweighted paths. Essential for level-order traversal, connected components, and shortest path in unweighted graphs.

Data structure: Queue (FIFO) — collections.deque for O(1) popleft
Time: O(V + E) where V = vertices, E = edges   Space: O(V) for queue and visited set
Guarantees shortest path: Yes, in unweighted graphs

Data structure: Call stack (recursive) or explicit stack (iterative)
Time: O(V + E)   Space: O(V) for visited set + O(V) recursion depth

Greedy & Backtracking Algorithms

Two complementary strategies — greedy makes the locally optimal choice at every step, while backtracking systematically explores and prunes all possibilities.

🏃

Greedy Algorithms

Make the locally optimal choice at each step, hoping it leads to a globally optimal solution. Greedy works when the problem has the greedy-choice property — no need to reconsider past decisions.

  • Activity / interval scheduling
  • Huffman coding
  • Dijkstra's shortest path
  • Fractional knapsack
  • Kruskal's & Prim's MST
🔙

Backtracking

Explore all candidates systematically using DFS, abandoning (pruning) paths that cannot lead to a valid solution. Generates all permutations, combinations, or valid configurations.

  • Permutations & combinations
  • Subsets (power set)
  • N-Queens problem
  • Sudoku solver
  • Word search in grid
Greedy — Interval Scheduling & Activity Selection

Select the maximum number of non-overlapping activities. The greedy strategy: always pick the activity that finishes earliest — this leaves the most room for future activities. Provably optimal via exchange argument.

Strategy: Sort by end time → greedily select next compatible activity
Time: O(n log n) for sorting   Space: O(1)
Proof of optimality: Exchange argument — swapping any chosen activity for the earliest-finishing one never worsens the solution
interval_scheduling.py
def max_activities(intervals): # Sort by end time (greedy choice: earliest finish first) intervals.sort(key=lambda x: x[1]) count = 1 last_end = intervals[0][1] for start, end in intervals[1:]: if start >= last_end: count += 1 last_end = end return count # Merge overlapping intervals (related pattern) def merge_intervals(intervals): intervals.sort(key=lambda x: x[0]) merged = [] for interval in intervals: if not merged or merged[-1][1] < interval[0]: merged.append(interval) else: merged[-1][1] = max(merged[-1][1], interval[1]) return merged # max_activities([(1,3),(2,5),(4,7),(6,9)]) → 2 # merge_intervals([(1,3),(2,6),(8,10)]) → [(1,6),(8,10)]
Backtracking Template — Permutations, Combinations, Subsets

The backtracking template: choose a candidate → recurse → undo the choice (backtrack). Three core variations: permutations (order matters, use all elements), combinations (order doesn't matter, choose k), and subsets (all possible selections).

Permutations: O(n!) — shrinking candidate list, no start index
Combinations: O(C(n,k)) — start index + length constraint
Subsets: O(2ⁿ) — collect at every node, not just leaves
backtracking_template.py
# Permutations — all orderings of elements def permutations(nums): result = [] def dfs(path, remaining): if not remaining: result.append(path[:]) return for i in range(len(remaining)): path.append(remaining[i]) dfs(path, remaining[:i] + remaining[i+1:]) path.pop() dfs([], nums) return result # Subsets — all possible selections (power set) def subsets(nums): result = [] def dfs(start, path): result.append(path[:]) for i in range(start, len(nums)): path.append(nums[i]) dfs(i + 1, path) path.pop() dfs(0, []) return result # Combinations — choose k from n def combinations(n, k): result = [] def dfs(start, path): if len(path) == k: result.append(path[:]) return for i in range(start, n + 1): path.append(i) dfs(i + 1, path) path.pop() dfs(1, []) return result # permutations([1,2,3]) → [[1,2,3],[1,3,2],...] # subsets([1,2,3]) → [[],[1],[1,2],[1,2,3],...] # combinations(4, 2) → [[1,2],[1,3],[1,4],[2,3],...]
N-Queens — Constraint-Based Backtracking

Place N queens on an N×N chessboard so that no two queens threaten each other. Place one queen per row, tracking attacked columns and diagonals with sets for O(1) conflict checking. The quintessential backtracking problem.

Constraint sets: cols, diag1 (row - col), diag2 (row + col)
Time: O(n!)   Space: O(n²) for board
Pruning: Skip any column or diagonal already under attack
n_queens.py
def solve_n_queens(n): result = [] board = [["."] * n for _ in range(n)] def backtrack(row, cols, diag1, diag2): if row == n: result.append(["".join(r) for r in board]) return for col in range(n): if col in cols or (row - col) in diag1 \ or (row + col) in diag2: continue board[row][col] = "Q" cols.add(col) diag1.add(row - col) diag2.add(row + col) backtrack(row + 1, cols, diag1, diag2) board[row][col] = "." cols.remove(col) diag1.remove(row - col) diag2.remove(row + col) backtrack(0, set(), set(), set()) return result # solve_n_queens(4) → 2 solutions
Sliding Window Technique — Variable and Fixed-Size Windows

Maintain a "window" over a contiguous portion of an array/string, expanding and shrinking it to satisfy constraints. Converts brute-force O(n²) subarray problems into O(n) single-pass solutions. The two main variants: fixed-size and variable-size windows.

Fixed window: Slide window of size k — add right, remove left each step
Variable window: Expand right until condition met, then shrink left to find minimum window
Time: O(n) — each element enters and leaves the window at most once

Break the problem into smaller independent subproblems, solve each recursively, then merge the results. Unlike DP, subproblems don't overlap. Merge sort, quick sort, and binary search are all divide and conquer. The key insight: subproblems are independent.

Three steps: 1) Divide into subproblems → 2) Conquer (solve recursively) → 3) Combine results
Master Theorem: T(n) = aT(n/b) + O(nᵈ) — determines complexity from recurrence
vs Dynamic Programming: D&C = non-overlapping subproblems. DP = overlapping subproblems.

Dynamic Programming

Break complex problems into overlapping subproblems — solve each once, store results, and build up to the final answer.

🧮

Core Concepts

  • Optimal substructure: Optimal solution contains optimal solutions to subproblems
  • Overlapping subproblems: Same subproblems are solved multiple times
  • Top-down (Memoization): Recursive with cache — easier to write, risk stack overflow
  • Bottom-up (Tabulation): Iterative with table — no recursion overhead, often more efficient
  • State reduction: Often only last 1-2 rows/values needed — optimize O(n²) space to O(n)
📋

Classic DP Patterns

  • 0/1 Knapsack: Take or skip items to maximize value within capacity
  • Unbounded Knapsack: Unlimited copies of each item (coin change)
  • Longest Common Subsequence: 2D DP on two strings
  • Longest Increasing Subsequence: O(n²) DP or O(n log n) with patience sorting
  • Matrix Chain Multiplication: Interval DP — optimal parenthesization
  • Grid paths: Unique paths, minimum path sum — fill row by row
  • State machine: Track multiple states (e.g. hold/sold/cooldown)
Memoization vs Tabulation — Top-Down vs Bottom-Up

The two approaches to dynamic programming. Memoization starts from the original problem and recursively breaks it down, caching results. Tabulation starts from the smallest subproblem and iteratively builds up. Both avoid redundant computation.

Memoization: Easier to code, only solves needed subproblems, but uses call stack (recursion limit)
Tabulation: No recursion overhead, predictable memory, but may solve unnecessary subproblems
Rule of thumb: Start with memoization for clarity, convert to tabulation for performance
fibonacci_dp.py
# Top-down: Memoization def fib_memo(n, memo={}): if n <= 1: return n if n not in memo: memo[n] = fib_memo(n - 1) + fib_memo(n - 2) return memo[n] # Bottom-up: Tabulation def fib_tab(n): if n <= 1: return n dp = [0] * (n + 1) dp[1] = 1 for i in range(2, n + 1): dp[i] = dp[i - 1] + dp[i - 2] return dp[n] # Space-optimized: Only keep last 2 values def fib_optimal(n): if n <= 1: return n a, b = 0, 1 for _ in range(2, n + 1): a, b = b, a + b return b print(fib_optimal(10)) # 55
0/1 Knapsack — Take or Skip Each Item

Given n items with weights and values, and a knapsack of capacity W, maximize total value without exceeding capacity. Each item can be taken at most once. The 1D optimization iterates weights in reverse to avoid using items twice.

Recurrence: dp[i][w] = max(dp[i-1][w], dp[i-1][w-wt[i]] + val[i])
Base case: dp[0][w] = 0 for all w
Time: O(n × W)   Space: O(W) with 1D optimization
knapsack_01.py
def knapsack(weights, values, capacity): n = len(weights) dp = [0] * (capacity + 1) for i in range(n): # Reverse to avoid reusing same item for w in range(capacity, weights[i] - 1, -1): dp[w] = max(dp[w], dp[w - weights[i]] + values[i]) return dp[capacity] # weights=[2,3,4,5], values=[3,4,5,6], capacity=8 → 10
Unbounded Knapsack / Coin Change — Reusable Items

When items can be reused, iterate weights forward instead of backward. The classic example is Coin Change: find the fewest coins to make a target amount. Each denomination can be used unlimited times.

Recurrence: dp[x] = min(dp[x], dp[x - coin] + 1) for each coin
0/1 vs Unbounded: Reverse loop = 0/1 (no reuse). Forward loop = unbounded (reuse allowed)
Time: O(amount × coins)   Space: O(amount)
coin_change.py
def coin_change(coins, amount): dp = [float("inf")] * (amount + 1) dp[0] = 0 for coin in coins: # Forward iteration — allows reuse of each coin for x in range(coin, amount + 1): dp[x] = min(dp[x], dp[x - coin] + 1) return dp[amount] if dp[amount] != float("inf") else -1 # coins=[1,5,11], amount=15 → 3 (5+5+5) # Greedy would pick 11+1+1+1+1 = 5 coins (wrong!)
Kadane's Algorithm — Maximum Subarray Sum

Find the contiguous subarray with the largest sum. At each position, decide whether to extend the current subarray or start fresh. The simplest and most elegant DP pattern — a single pass with O(1) space.

Recurrence: current = max(nums[i], current + nums[i])
Decision: Extend current subarray OR start new subarray from position i
Time: O(n)   Space: O(1)
kadanes_algorithm.py
def max_subarray(nums): max_sum = nums[0] current = nums[0] for i in range(1, len(nums)): current = max(nums[i], current + nums[i]) max_sum = max(max_sum, current) return max_sum # [-2, 1, -3, 4, -1, 2, 1, -5, 4] → 6 # Best subarray: [4, -1, 2, 1]
Longest Increasing Subsequence (LIS) — O(n²) and O(n log n)

Find the length of the longest strictly increasing subsequence. The O(n²) approach compares each pair; the O(n log n) approach maintains a "tails" array and uses binary search (patience sorting) — one of the most elegant algorithm optimizations.

O(n²): dp[i] = max(dp[j] + 1) for all j < i where nums[j] < nums[i]
O(n log n): tails[i] = smallest possible tail element of an increasing subsequence of length i+1
Key insight: Keeping smaller tails maximizes chances of extending the subsequence
longest_increasing_subsequence.py
# O(n²) DP approach def lis_dp(nums): dp = [1] * len(nums) for i in range(1, len(nums)): for j in range(i): if nums[j] < nums[i]: dp[i] = max(dp[i], dp[j] + 1) return max(dp) # O(n log n) patience sorting approach from bisect import bisect_left def lis_optimal(nums): tails = [] for num in nums: pos = bisect_left(tails, num) if pos == len(tails): tails.append(num) # Extend else: tails[pos] = num # Replace with smaller tail return len(tails) # [10, 9, 2, 5, 3, 7, 101, 18] → 4 # LIS: [2, 3, 7, 101]
Longest Common Subsequence (LCS) — 2D String DP

Find the longest subsequence common to two strings. This is the canonical 2D DP problem: build a table where dp[i][j] represents the LCS of the first i characters of text1 and first j characters of text2. Used in diff tools and bioinformatics.

Recurrence: If chars match: dp[i][j] = dp[i-1][j-1] + 1. Else: max(dp[i-1][j], dp[i][j-1])
Time: O(m × n)   Space: O(m × n), reducible to O(min(m, n))
longest_common_subsequence.py
def lcs(text1, text2): m, n = len(text1), len(text2) dp = [[0] * (n + 1) for _ in range(m + 1)] for i in range(1, m + 1): for j in range(1, n + 1): if text1[i - 1] == text2[j - 1]: dp[i][j] = dp[i - 1][j - 1] + 1 else: dp[i][j] = max(dp[i - 1][j], dp[i][j - 1]) return dp[m][n] # lcs("abcde", "ace") → 3 ("ace") # lcs("abc", "def") → 0
Grid DP — Counting Paths and Minimum Cost

Navigate a 2D grid (moving right/down) and compute path counts or minimum costs. Each cell depends only on the cell above and to the left — the most intuitive 2D DP pattern. Extends naturally to grids with obstacles.

Unique Paths: dp[i][j] = dp[i-1][j] + dp[i][j-1]
Min Path Sum: dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])
Base case: First row and first column filled directly   Space: O(n) with 1D optimization
grid_dp.py
# Count unique paths from top-left to bottom-right def unique_paths(m, n): dp = [[1] * n for _ in range(m)] for i in range(1, m): for j in range(1, n): dp[i][j] = dp[i - 1][j] + dp[i][j - 1] return dp[m - 1][n - 1] # Minimum path sum through a cost grid def min_path_sum(grid): m, n = len(grid), len(grid[0]) for i in range(1, m): grid[i][0] += grid[i - 1][0] for j in range(1, n): grid[0][j] += grid[0][j - 1] for i in range(1, m): for j in range(1, n): grid[i][j] += min(grid[i-1][j], grid[i][j-1]) return grid[m - 1][n - 1] # unique_paths(3, 7) → 28 # unique_paths(3, 2) → 3
Linear DP — Include/Exclude Pattern

At each step, decide to include or exclude the current element. The recurrence relates to the previous 1-2 states. This pattern appears in house robber, climbing stairs, paint fence, and many other sequence optimization problems.

Recurrence: dp[i] = max(val[i] + dp[i-2], dp[i-1])
Key insight: Only two states needed — "best including i" and "best excluding i"
Time: O(n)   Space: O(1) with two variables

# Maximum sum of non-adjacent elements def max_non_adjacent(nums): if not nums: return 0 if len(nums) <= 2: return max(nums) prev2 = nums[0] prev1 = max(nums[0], nums[1]) for i in range(2, len(nums)): current = max(nums[i] + prev2, prev1) prev2 = prev1 prev1 = current return prev1 # [2, 7, 9, 3, 1] → 12 (pick indices 0, 2, 4: 2+9+1) # [1, 2, 3, 1] → 4 (pick indices 0, 2: 1+3)

Graph Algorithms

From shortest paths to cycle detection to minimum spanning trees — the essential algorithms for working with connected data.

Dijkstra's Algorithm — Shortest Path in Weighted Graphs

Find the shortest path from a source node to all other nodes in a graph with non-negative edge weights. Uses a priority queue (min-heap) to always process the closest unvisited node first — a greedy approach that's provably optimal.

Data structure: Min-heap / priority queue for efficient extraction of minimum distance node
Time: O((V + E) log V) with binary heap   Space: O(V)
Limitation: Does NOT work with negative edge weights — use Bellman-Ford for that
dijkstra.py
import heapq def dijkstra(graph, source): # graph: {node: [(neighbor, weight), ...]} dist = {node: float("inf") for node in graph} dist[source] = 0 heap = [(0, source)] while heap: d, u = heapq.heappop(heap) if d > dist[u]: continue # Skip stale entries for v, weight in graph[u]: new_dist = d + weight if new_dist < dist[v]: dist[v] = new_dist heapq.heappush(heap, (new_dist, v)) return dist # Example graph graph = { "A": [("B", 1), ("C", 4)], "B": [("C", 2), ("D", 5)], "C": [("D", 1)], "D": [] } print(dijkstra(graph, "A")) # {'A': 0, 'B': 1, 'C': 3, 'D': 4}
Bellman-Ford Algorithm — Handles Negative Weights

Find shortest paths from a single source, even with negative edge weights. Relax all edges V-1 times — after V-1 iterations, all shortest paths are found. A V-th iteration can detect negative-weight cycles.

Strategy: Relax every edge V-1 times — each iteration guarantees one more vertex is finalized
Time: O(V × E)   Space: O(V)
Negative cycle detection: If any edge can still be relaxed after V-1 iterations, a negative cycle exists
bellman_ford.py
def bellman_ford(vertices, edges, source): # edges: list of (u, v, weight) dist = {v: float("inf") for v in vertices} dist[source] = 0 # Relax all edges V-1 times for _ in range(len(vertices) - 1): for u, v, w in edges: if dist[u] + w < dist[v]: dist[v] = dist[u] + w # Check for negative-weight cycles for u, v, w in edges: if dist[u] + w < dist[v]: raise ValueError("Negative cycle detected") return dist # vertices = ["A","B","C","D"] # edges = [("A","B",1),("B","C",-2),("A","C",4),("C","D",3)] # bellman_ford(vertices, edges, "A") # → {'A': 0, 'B': 1, 'C': -1, 'D': 2}
Topological Sort — Ordering Directed Acyclic Graphs

Produce a linear ordering of vertices such that for every edge (u, v), u comes before v. Only possible on DAGs (Directed Acyclic Graphs). Two approaches: Kahn's algorithm (BFS with in-degree tracking) and DFS-based (post-order reversal).

Kahn's (BFS): Start with nodes of in-degree 0 → process → decrement neighbor in-degrees → repeat
DFS-based: DFS post-order, then reverse the result
Time: O(V + E)   Cycle detection: If result has fewer than V nodes, a cycle exists
topological_sort.py
from collections import deque, defaultdict # Kahn's Algorithm (BFS-based) def topological_sort_bfs(num_nodes, edges): graph = defaultdict(list) in_degree = [0] * num_nodes for u, v in edges: graph[u].append(v) in_degree[v] += 1 # Start with all zero in-degree nodes queue = deque(i for i in range(num_nodes) if in_degree[i] == 0) order = [] while queue: node = queue.popleft() order.append(node) for neighbor in graph[node]: in_degree[neighbor] -= 1 if in_degree[neighbor] == 0: queue.append(neighbor) if len(order) != num_nodes: return [] # Cycle detected! return order # 4 courses: 0→1, 0→2, 1→3, 2→3 # topological_sort_bfs(4, [(0,1),(0,2),(1,3),(2,3)]) # → [0, 1, 2, 3] or [0, 2, 1, 3]
Cycle Detection — Directed and Undirected Graphs

Detect if a graph contains a cycle. For directed graphs, use DFS with three-color marking (white/gray/black): a back edge to a gray node means a cycle. For undirected graphs, track the parent — visiting a node that's already visited and isn't the parent means a cycle.

Directed: DFS with states — WHITE (unvisited), GRAY (in progress), BLACK (done)
Undirected: DFS tracking parent — or use Union-Find
Time: O(V + E) for both
cycle_detection.py
# Directed graph: DFS with 3-color marking def has_cycle_directed(graph, num_nodes): WHITE, GRAY, BLACK = 0, 1, 2 color = [WHITE] * num_nodes def dfs(node): color[node] = GRAY for neighbor in graph[node]: if color[neighbor] == GRAY: return True # Back edge → cycle! if color[neighbor] == WHITE and dfs(neighbor): return True color[node] = BLACK return False return any(color[i] == WHITE and dfs(i) for i in range(num_nodes)) # Undirected graph: DFS tracking parent def has_cycle_undirected(graph, num_nodes): visited = set() def dfs(node, parent): visited.add(node) for neighbor in graph[node]: if neighbor not in visited: if dfs(neighbor, node): return True elif neighbor != parent: return True # Visited & not parent → cycle! return False return any(i not in visited and dfs(i, -1) for i in range(num_nodes))
Union-Find (Disjoint Set Union) — Connected Components

Efficiently track which elements belong to the same group. Two operations: find (which set does x belong to?) and union (merge two sets). With path compression and union by rank, both operations run in nearly O(1) amortized time.

Path compression: During find, point each node directly to the root — flattens the tree
Union by rank: Attach smaller tree under larger tree's root
Time: O(α(n)) per operation ≈ O(1) amortized   Space: O(n)
union_find.py
class UnionFind: def __init__(self, n): self.parent = list(range(n)) self.rank = [1] * n self.components = n def find(self, x): if x != self.parent[x]: self.parent[x] = self.find(self.parent[x]) return self.parent[x] def union(self, a, b): ra, rb = self.find(a), self.find(b) if ra == rb: return False # Already connected if self.rank[ra] < self.rank[rb]: ra, rb = rb, ra self.parent[rb] = ra self.rank[ra] += self.rank[rb] self.components -= 1 return True def connected(self, a, b): return self.find(a) == self.find(b) # Example: cycle detection in undirected graph uf = UnionFind(5) edges = [(0,1), (1,2), (2,3), (3,0)] for u, v in edges: if not uf.union(u, v): print(f"Cycle: edge ({u},{v})") # "Cycle: edge (3,0)" — nodes 3 and 0 already connected

Trie (Prefix Tree) — Efficient String Operations

A tree where each node represents a character, and paths from root to nodes represent prefixes. Tries enable O(L) insert, search, and prefix matching (where L = word length) — independent of how many words are stored. Essential for autocomplete, spell checking, and IP routing.

Insert / Search / Prefix: O(L) where L = word length
Space: O(N × L × alphabet_size) worst case, but shared prefixes save memory
When to use: Prefix matching, autocomplete, word dictionaries, IP routing tables
trie.py
class TrieNode: def __init__(self): self.children = {} self.is_end = False class Trie: def __init__(self): self.root = TrieNode() def insert(self, word): node = self.root for char in word: if char not in node.children: node.children[char] = TrieNode() node = node.children[char] node.is_end = True def search(self, word): node = self._find(word) return node is not None and node.is_end def starts_with(self, prefix): return self._find(prefix) is not None def _find(self, prefix): node = self.root for char in prefix: if char not in node.children: return None node = node.children[char] return node # trie = Trie() # trie.insert("apple") # trie.search("apple") → True # trie.search("app") → False # trie.starts_with("app") → True
Heap / Priority Queue — Efficient Min/Max Access

A complete binary tree where every parent is smaller (min-heap) or larger (max-heap) than its children. Heaps provide O(1) access to the min/max and O(log n) insertion and extraction. Python's heapq module implements a min-heap.

Operations: Insert O(log n) | Extract min/max O(log n) | Peek O(1) | Heapify O(n)
Python: heapq is a min-heap — negate values for max-heap behavior
Applications: Dijkstra's, K-th largest, median from stream, task scheduling

Binary Indexed Tree (Fenwick Tree) — Range Sum Queries

A compact tree structure for efficiently computing prefix sums and performing point updates on an array. Uses clever bit manipulation: i & -i extracts the lowest set bit, which determines each node's range of responsibility.

Update (point): O(log n) — add value at index, propagate up via i += i & -i
Query (prefix sum): O(log n) — accumulate down via i -= i & -i
Space: O(n) — much simpler to implement than a Segment Tree
fenwick_tree.py
class FenwickTree: def __init__(self, n): self.n = n self.tree = [0] * (n + 1) def update(self, i, delta): # Add delta at index i (1-indexed) while i <= self.n: self.tree[i] += delta i += i & (-i) # Move to parent def prefix_sum(self, i): # Sum from index 1 to i total = 0 while i > 0: total += self.tree[i] i -= i & (-i) # Move to predecessor return total def range_sum(self, left, right): return self.prefix_sum(right) - self.prefix_sum(left - 1) # ft = FenwickTree(5) # for i, v in enumerate([1,3,5,7,9], 1): # ft.update(i, v) # ft.range_sum(2, 4) → 15 (3+5+7) # ft.update(3, 2) # Add 2 to index 3 # ft.range_sum(2, 4) → 17 (3+7+7)
Segment Tree — Range Queries and Updates

A binary tree where each node stores aggregate information (sum, min, max) over a range of elements. Supports both range queries and range updates in O(log n). More flexible than a Fenwick tree — can handle min/max queries, not just sums.

Build: O(n)   Query: O(log n)   Update: O(log n)
Space: O(4n) for array-based representation
vs Fenwick: Segment tree handles min/max queries and range updates; Fenwick is simpler for prefix sums
segment_tree.py
class SegmentTree: def __init__(self, arr): self.n = len(arr) self.tree = [0] * (4 * self.n) self._build(arr, 1, 0, self.n - 1) def _build(self, arr, node, start, end): if start == end: self.tree[node] = arr[start] return mid = (start + end) // 2 self._build(arr, 2*node, start, mid) self._build(arr, 2*node+1, mid+1, end) self.tree[node] = self.tree[2*node] + self.tree[2*node+1] def query(self, l, r): return self._query(1, 0, self.n-1, l, r) def _query(self, node, start, end, l, r): if r < start or end < l: return 0 if l <= start and end <= r: return self.tree[node] mid = (start + end) // 2 return (self._query(2*node, start, mid, l, r) + self._query(2*node+1, mid+1, end, l, r)) def update(self, idx, val): self._update(1, 0, self.n-1, idx, val) def _update(self, node, start, end, idx, val): if start == end: self.tree[node] = val return mid = (start + end) // 2 if idx <= mid: self._update(2*node, start, mid, idx, val) else: self._update(2*node+1, mid+1, end, idx, val) self.tree[node] = self.tree[2*node] + self.tree[2*node+1] # st = SegmentTree([1, 3, 5, 7, 9]) # st.query(1, 3) → 15 (3+5+7) # st.update(2, 10) # st.query(1, 3) → 20 (3+10+7)
Reservoir Sampling — Random Sampling from Streams

Select k random items from a stream of unknown length, where each element has an equal probability of being chosen. The algorithm processes elements one at a time and doesn't need to know the total number of elements in advance.

Algorithm: Fill reservoir with first k items. For the i-th item (i > k), replace a random reservoir item with probability k/i
Time: O(n) single pass   Space: O(k)
Guarantee: Each element has exactly k/n probability of being in the final sample
reservoir_sampling.py
import random def reservoir_sample(stream, k): reservoir = [] for i, item in enumerate(stream): if i < k: reservoir.append(item) else: # Replace element j with probability k/(i+1) j = random.randint(0, i) if j < k: reservoir[j] = item return reservoir # Select 3 random items from a large stream stream = range(1000000) sample = reservoir_sample(stream, 3) # Each element has exactly 3/1000000 probability
