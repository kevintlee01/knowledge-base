Chapter 1

What Are Microservices?

Microservices is an architectural style that structures an application as a collection of small, autonomous, loosely-coupled services, each responsible for a specific business capability.

📐

Definition

A microservices architecture decomposes a monolithic application into a suite of independently deployable services. Each service runs its own process, communicates via lightweight protocols (typically HTTP/REST or messaging), and can be developed, deployed, and scaled independently by a small team.

The term was popularized around 2011–2014 by thought leaders including Martin Fowler, James Lewis, and Sam Newman, though the ideas trace back to Service-Oriented Architecture (SOA) and Unix philosophy.

🎯

Key Characteristics

  • Single Responsibility: Each service encapsulates one business domain
  • Autonomous Deployment: Services deploy independently
  • Decentralized Governance: Teams choose their own tech stacks
  • Polyglot Persistence: Different databases per service
  • Fault Isolation: Failure in one service doesn't cascade
  • Built for Change: Easy to replace or rewrite individual services
  • Organized Around Business Capabilities: Aligned with Conway's Law
  • Smart Endpoints, Dumb Pipes: Logic in services, not middleware

Chapter 2

Core Principles

The foundational principles that govern the design and operation of microservices architectures.

🔗

Loose Coupling

Services interact through well-defined interfaces with minimal dependencies on each other's internal implementation. Changes to one service should not require changes in others. This is achieved through API contracts, event-driven communication, and avoiding shared databases.

🧩

High Cohesion

Related functionality is grouped within a single service (Bounded Context from DDD). Each service should contain everything it needs to fulfill its business capability. If you frequently change two services together, they should probably be merged.

🏗️

Domain-Driven Design

Services are modeled around business domains using DDD concepts: Bounded Contexts define service boundaries, Aggregates ensure transactional consistency, Domain Events enable cross-service communication, and Ubiquitous Language aligns code with business terminology.

💻 Domain model — Order aggregate with domain events
// domain/order.ts — Pure domain logic, zero infrastructure imports class Order { private events: DomainEvent[] = []; constructor( public readonly id: string, public status: OrderStatus, public readonly userId: string, public readonly items: OrderItem[], ) {} static create(userId: string, items: OrderItem[]): Order { if (items.length === 0) throw new DomainError('Order must have at least one item'); const order = new Order(uuid(), OrderStatus.PENDING, userId, items); order.addEvent(new OrderCreated({ orderId: order.id, userId, items })); return order; } confirm(paymentId: string) { if (this.status !== OrderStatus.PENDING) throw new DomainError(`Cannot confirm order in ${this.status} state`); this.status = OrderStatus.CONFIRMED; this.addEvent(new OrderConfirmed({ orderId: this.id, paymentId })); } cancel(reason: string) { if (this.status === OrderStatus.SHIPPED) throw new DomainError('Cannot cancel shipped order'); this.status = OrderStatus.CANCELLED; this.addEvent(new OrderCancelled({ orderId: this.id, reason })); } getUncommittedEvents(): DomainEvent[] { return [...this.events]; } private addEvent(e: DomainEvent) { this.events.push(e); } } enum OrderStatus { PENDING, CONFIRMED, SHIPPED, DELIVERED, CANCELLED }

📦

Single Responsibility

Each microservice does one thing well. It owns its data and business logic for a single business capability. This keeps services small (typically 100–5000 LOC), understandable, and independently replaceable.

🛡️

Design for Failure

In distributed systems, failure is inevitable. Services must implement circuit breakers, retries with exponential backoff, bulkheads, timeouts, and graceful degradation. The system should continue functioning even when individual services are unavailable.

⚙️

Infrastructure Automation

CI/CD pipelines, infrastructure as code (IaC), containerization, and orchestration are essential. Automation reduces operational overhead of managing many services. You can't effectively run microservices without automated testing, deployment, and monitoring.

🔄

Evolutionary Design

Services should be designed to be disposable and replaceable. Prefer evolutionary improvements over big-bang rewrites. Use the Strangler Fig pattern to incrementally replace monolithic components. Version APIs to prevent breaking changes.

👥

Decentralized Governance

Each team chooses the best technology for their service. One service might use Node.js with MongoDB while another uses Java with PostgreSQL. This autonomy enables teams to innovate and optimize for their specific domain needs.

📊

Observability First

Every service must emit metrics, logs, and traces. Distributed tracing (OpenTelemetry), centralized logging (ELK/Loki), and metrics dashboards (Prometheus/Grafana) are non-negotiable. You cannot debug what you cannot observe.

        🛒 Place Order — Happy Path

        1. Client → POST /api/orders → API Gateway
        2. Gateway validates JWT, checks rate limit (token bucket: 100 req/s), routes to Order Service
        3. Order Service validates payload (JSON Schema), creates order aggregate
        4. Order Service → gRPC → Inventory Service: ReserveItems(orderId, items[])
        5. Inventory Service checks stock using SELECT ... FOR UPDATE (row-level locking), reserves items
        6. Order Service → gRPC → Payment Service: ProcessPayment(orderId, amount, paymentMethod)
        7. Payment Service calls PSP (Stripe/Adyen) via idempotent request (idempotency key = orderId)
        8. Payment confirmed → Order Service publishes OrderConfirmed event to Kafka topic orders.confirmed
        9. Notification Service consumes event → sends email/push via async worker pool
        10. Shipping Service consumes event → creates shipment label → publishes ShipmentCreated
        11. Response: 201 Created with order ID and estimated delivery → Client

        🔄 Saga Rollback — Payment Fails

        1. Steps 1–5 succeed (order created, inventory reserved)
        2. Payment Service returns 402 Payment Required (card declined)
        3. Saga Orchestrator initiates compensating transactions:
          → ReleaseInventory(orderId, items[]) — Inventory Service rolls back reservation
          → CancelOrder(orderId) — Order Service sets status to CANCELLED
        4. Orchestrator publishes OrderFailed event to orders.failed topic
        5. Notification Service consumes event → notifies customer of failure
        6. Key invariant: Each compensating transaction is idempotent — safe to retry on partial failure
        7. Saga state machine transitions: STARTED → RESERVING → PAYING → COMPENSATING → FAILED
        8. All state transitions stored in saga log table for auditability

        ⚡ Circuit Breaker in Action

        Config: failureRateThreshold: 50%, slidingWindowSize: 10, waitDurationInOpenState: 30s

        CLOSED → 6/10 calls to Payment Service fail (60% > 50%) → circuit OPENS
        OPEN → All calls fail-fast with 503 Service Unavailable for 30 seconds
          → Fallback: return cached response or queue for retry
        HALF-OPEN → After 30s, allow 3 probe requests through
          → If 2/3 succeed → circuit CLOSES (reset counters)
          → If 2/3 fail → circuit re-OPENS (restart wait timer)

        Metrics emitted: circuit_breaker_state{service="payment"}, circuit_breaker_failure_rate, circuit_breaker_calls_total{outcome="success|failure|not_permitted"}

        📨 Event-Driven Async Flow

        Producer (Order Service):
        1. Write order to DB + outbox row in same transaction (Transactional Outbox)
        2. Debezium CDC reads PostgreSQL WAL → publishes to Kafka orders.events
        3. Message format: CloudEvents v1.0 envelope with type: com.shop.order.created

        Consumer (Inventory Service):
        4. Consumer group inventory-svc reads from partition (key = orderId for ordering)
        5. Process event with at-least-once semantics → deduplication via idempotency table
        6. Commit offset only after successful processing
        7. Dead Letter Queue (DLQ) for poison messages after 3 retries

        Kafka Deep Internals

        Partitioning & Consumer Groups

        Topic: orders.events — 12 partitions, replication-factor 3

        Partition key: orderId (hash mod 12) — guarantees ordering per order

        Consumer group: inventory-svc with 4 instances
        → Each instance gets 3 partitions (12/4 = 3)
        → If an instance dies, partitions rebalance to remaining 3 instances
        → session.timeout.ms=30000, heartbeat.interval.ms=10000

        Rebalance protocol: Cooperative Sticky Assignor (KIP-429) for incremental rebalancing — avoids stop-the-world during deploys

        Offset management: Consumer commits offset after processing. enable.auto.commit=false for at-least-once. Sync commit on batch boundaries.

        Exactly-Once & Transactions

        Producer idempotence: enable.idempotence=true
        → Broker deduplicates by (PID, sequence number) per partition

        Transactional producer:
        producer.initTransactions()
        producer.beginTransaction()
        producer.send(record1)
        producer.send(record2)
        producer.sendOffsetsToTransaction(offsets, consumerGroupId)
        producer.commitTransaction()

        Consumer: isolation.level=read_committed — only sees committed transactional messages

        Compacted topics: cleanup.policy=compact for latest-state-per-key semantics (e.g., user profile snapshots)
        Performance tuning: batch.size=65536 (64KB), linger.ms=10 (batch up to 10ms for throughput), compression.type=zstd (best ratio), buffer.memory=67108864 (64MB). Consumer: fetch.min.bytes=1024, max.poll.records=500. Monitoring: Track consumer lag via kafka_consumer_group_lag metric. Alert when lag exceeds SLO (e.g., > 10,000 messages for > 5 min).

        Service Mesh Architecture (Istio)

        Data Plane — Envoy Sidecar

        Every pod gets an Envoy proxy sidecar injected via istio-init init container (iptables redirect all traffic through proxy).

        Per-request path:
        1. App sends request to payment-svc:8080
        2. iptables redirects to local Envoy (127.0.0.1:15001)
        3. Envoy performs: mTLS handshake, load balancing (round-robin/least-conn/random), retry (2 attempts, 5xx only), timeout (3s default), circuit breaking
        4. Envoy on destination pod terminates mTLS, forwards to app on 127.0.0.1:8080

        Observability automatically added:
        → Distributed tracing headers (B3/W3C) propagated
        → Metrics: istio_requests_total, istio_request_duration_milliseconds
        → Access logs with upstream/downstream details

        Control Plane — istiod

        istiod = Pilot + Citadel + Galley merged into single binary

        Pilot: Converts high-level routing rules (VirtualService, DestinationRule) into Envoy xDS configuration. Pushes config to all sidecars via gRPC streaming.

        Citadel: Certificate authority for mTLS. Issues SPIFFE identity certs (spiffe://cluster.local/ns/default/sa/order-service). Auto-rotates every 24h.

        Traffic management examples:
        → VirtualService: route 90% traffic to v1, 10% to v2 (canary)
        → DestinationRule: circuit breaker config (outlierDetection)
        → AuthorizationPolicy: only allow payment-svc to call fraud-svc
        → PeerAuthentication: enforce STRICT mTLS namespace-wide

        Design Patterns

        Battle-tested patterns for solving common challenges in microservices architectures.

        Decomposition Patterns

        Decompose by Business Capability

        Identify business capabilities (what the org does) and create one service per capability. Example: Order Management, Inventory Management, Customer Management. Aligns with Conway's Law — team boundaries mirror service boundaries.

        Decompose by Subdomain (DDD)

        Use Domain-Driven Design to identify subdomains: Core (competitive advantage), Supporting (necessary but not differentiating), Generic (commodity functionality). Each bounded context becomes a service.

        Strangler Fig Pattern

        Incrementally migrate from monolith to microservices. Place a facade in front of the monolith, gradually route traffic to new services. Old functionality is "strangled" as new services replace it. Minimizes migration risk.

        Sidecar Pattern

        Deploy helper components alongside each service (as a separate process/container). The sidecar handles cross-cutting concerns: logging, monitoring, networking, security. Central to service mesh implementations like Istio and Linkerd.

        Integration Patterns

        API Gateway

        Single entry point for all clients. Routes requests to appropriate services, handles cross-cutting concerns (auth, rate limiting, caching). Can perform request aggregation — combining multiple service responses into one client response.

        • Request routing and composition
        • Protocol translation (REST → gRPC)
        • Response caching and transformation
        • Authentication and authorization

        Backend for Frontend (BFF)

        Dedicated API gateway per client type (web, mobile, IoT). Each BFF optimizes the API for its client's needs — different response formats, aggregation strategies, and error handling. Prevents a one-size-fits-all API.

        Saga Pattern

        Manages distributed transactions across services without 2PC (two-phase commit). Two types:

        • Choreography: Each service publishes events that trigger the next step. Decentralized but complex to debug.
        • Orchestration: A central coordinator directs the saga steps. Easier to understand but creates a single point of coordination.

        Compensating transactions handle rollback on failure.

        💻 Saga orchestrator — distributed transaction coordination
        // application/saga-orchestrator.ts — Orchestration-based saga class PlaceOrderSaga { constructor( private orderRepo: OrderRepository, private inventoryClient: InventoryClient, private paymentClient: PaymentClient, private eventPublisher: EventPublisher, ) {} async execute(cmd: CreateOrderCommand) { const order = Order.create(cmd.userId, cmd.items); await this.orderRepo.save(order); try { // Step 1: Reserve inventory const reservation = await this.inventoryClient.reserve(order.items); // Step 2: Process payment const payment = await this.paymentClient.charge( order.id, order.totalAmount(), 'USD' ); // Step 3: Confirm order + publish event order.confirm(payment.paymentId); await this.orderRepo.save(order); await this.eventPublisher.publishAll(order.getUncommittedEvents()); return order; } catch (err) { // Compensating transactions — undo in reverse order await this.inventoryClient.release(order.id).catch(logCompensationError); order.cancel(err.message); await this.orderRepo.save(order); await this.eventPublisher.publish(new OrderFailed({ orderId: order.id, reason: err.message })); throw err; } } }

        Event Sourcing + CQRS

        Event Sourcing: Store state changes as an immutable sequence of events rather than current state. Enables complete audit trail, temporal queries, and event replay.

        CQRS (Command Query Responsibility Segregation): Separate read and write models. Write side appends events, read side projects optimized views. Enables independent scaling of reads vs writes.

        Shared Database (Anti-pattern)

        Multiple services sharing a single database. While simpler initially, it creates tight coupling — schema changes affect all services, and cross-service transactions are tempting. Generally considered an anti-pattern but sometimes pragmatic for migrations.

        Transactional Outbox

        Solves the dual-write problem: how to atomically update a database and publish an event. The service writes to an outbox table in the same transaction as the business data. A separate process (Debezium/CDC) reads the outbox and publishes events to the message broker.

        API Composition

        Implements queries that span multiple services by invoking each service's API and combining results in-memory. The API Gateway or a dedicated composer service performs the aggregation. Works well for simple queries but can have latency and consistency challenges.

        Resilience Patterns

        Circuit Breaker

        Prevents cascading failures by monitoring failure rates. Three states:

        • Closed: Normal operation, requests pass through
        • Open: Failure threshold exceeded, requests fail fast
        • Half-Open: After timeout, allows test requests to check recovery

        Implementations: Resilience4j, Hystrix (deprecated), Polly (.NET), gobreaker (Go).

        💻 Node.js — Opossum Circuit Breaker
        // Node.js — Opossum circuit breaker (most popular CB for Node.js) const CircuitBreaker = require('opossum'); async function callPaymentService(orderId, amount) { const res = await fetch(`http://payment-svc/charge`, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ orderId, amount }), }); if (!res.ok) throw new Error(`Payment failed: ${res.status}`); return res.json(); } const breaker = new CircuitBreaker(callPaymentService, { timeout: 5000, // 5s per request errorThresholdPercentage: 50, // open at 50% failure rate resetTimeout: 30000, // try again after 30s volumeThreshold: 10, // min 10 requests before tripping }); // Fallback when circuit is open breaker.fallback((orderId) => ({ status: 'pending', message: 'Payment queued for retry', })); // Emit Prometheus metrics breaker.on('success', () => circuitMetrics.inc({ outcome: 'success' })); breaker.on('failure', () => circuitMetrics.inc({ outcome: 'failure' })); breaker.on('open', () => logger.warn('Circuit OPEN — payment-svc')); // Usage in route handler app.post('/api/orders/:id/pay', async (req, res) => { const result = await breaker.fire(req.params.id, req.body.amount); res.json(result); });

        Bulkhead

        Isolates resources so a failure in one component doesn't exhaust resources for others. Like bulkheads in a ship prevent flooding of all compartments. Implementation: dedicated thread pools/connection pools per downstream service, or process-level isolation via containers.

        Retry with Backoff

        Automatically retry failed requests with increasing delays. Prevents overwhelming a recovering service. Use exponential backoff with jitter to avoid thundering herd problems. Always set a maximum retry count and use idempotent operations.

        Timeout & Fallback

        Every external call must have a timeout. Without timeouts, a slow downstream service can cause thread pool exhaustion upstream. Combine with fallbacks: return cached data, default values, or degraded functionality when the primary path fails.

        Observability Patterns

        Distributed Tracing

        Track requests as they flow across services. Each request gets a unique trace ID propagated through all service calls. Spans represent individual operations within a trace. Essential for debugging latency and understanding request paths.

        • OpenTelemetry: The unified standard for traces, metrics, and logs
        • Jaeger / Zipkin: Trace collection and visualization
        • Context Propagation: W3C TraceContext, B3 headers

        Log Aggregation

        Centralize logs from all services into one searchable system. Each log entry must include correlation IDs for cross-service request tracking. Structured logging (JSON) enables efficient querying.

        • ELK Stack: Elasticsearch, Logstash, Kibana
        • Loki + Grafana: Lightweight, label-based log aggregation
        • Fluentd / Fluent Bit: Log collection and forwarding

        Communication Patterns

        How services talk to each other — synchronous vs asynchronous, protocols, and messaging strategies.

        🔵 Synchronous Communication

        Request-response pattern. The caller waits for a response. Simple to understand but creates temporal coupling — both services must be available simultaneously.

        REST / HTTP

        • Most common protocol for microservices APIs
        • Uses standard HTTP methods: GET, POST, PUT, PATCH, DELETE
        • JSON or XML payloads, OpenAPI/Swagger for documentation
        • Stateless, cacheable, uses standard HTTP status codes
        • Content negotiation via Accept/Content-Type headers
        • HATEOAS for discoverability (rarely used in practice)

        gRPC

        • High-performance RPC framework by Google
        • Protocol Buffers (protobuf) for serialization — 3-10× faster than JSON
        • HTTP/2 with multiplexing, bidirectional streaming
        • Strongly typed contracts via .proto files
        • Code generation for 12+ languages
        • Best for internal service-to-service communication

        GraphQL

        • Query language for APIs — clients specify exactly what data they need
        • Single endpoint, eliminates over/under-fetching
        • Schema stitching/federation for composing across services
        • Best suited for the API Gateway / BFF layer

        🟢 Asynchronous Communication

        Fire-and-forget or pub/sub patterns. Services are temporally decoupled — the sender doesn't wait for processing. Higher fault tolerance but more complex to debug.

        Message Brokers

        • Apache Kafka: Distributed event streaming platform. Append-only log, consumer groups, exactly-once semantics. Best for high-throughput event streaming and event sourcing.
        • RabbitMQ: AMQP message broker. Exchanges, queues, routing keys. Best for task queues and complex routing scenarios.
        • AWS SQS/SNS: Managed queuing (SQS) and pub/sub (SNS) services. Zero infrastructure management.
        • NATS: Lightweight, high-performance messaging. JetStream for persistence. Best for cloud-native microservices.
        • Redis Streams: Built on Redis, low-latency stream processing.

        Event-Driven Patterns

        • Event Notification: Lightweight events trigger further processing
        • Event-Carried State Transfer: Events carry complete state
        • Domain Events: Business-meaningful events (OrderPlaced, PaymentProcessed)
        • Integration Events: Cross-boundary communication events

        Eventual Consistency

        Async communication typically means eventual consistency. Data across services is not immediately consistent but converges over time. Design UIs and business processes to accommodate this reality.

        Data Management

        Strategies for managing data in a distributed microservices environment.

        Database per Service — The Fundamental Rule

        Each microservice has exclusive ownership of its data store. Other services can only access data through the owning service's API. This ensures:

        • Loose coupling: Schema changes don't propagate across services
        • Independent scalability: Scale databases based on each service's needs
        • Technology freedom: Choose the best database for each use case
        • Independent deployability: Database migrations don't require coordinated deployments

        The trade-off: cross-service queries, distributed transactions, and data consistency become significantly more complex.

        💻 Prisma schema — Database per service with migrations

        Event Sourcing

        Instead of storing current state, store the full history of state-changing events in an append-only event store.

        💻 Traditional: Store current state
        // Traditional: Store current state Order { id: 123, status: "shipped", total: 59.99 } // Event Sourcing: Store event history OrderCreated { id: 123, items: [...], total: 59.99 } PaymentReceived { id: 123, amount: 59.99 } OrderShipped { id: 123, tracking: "1Z999..." }

        Benefits: Complete audit trail, temporal queries (state at any point in time), event replay for rebuilding state, debugging, and analytics. Challenges: Event schema evolution, large event stores, eventual consistency in projections.

        CQRS — Command Query Responsibility Segregation

        Separate the write model (commands) from the read model (queries). The command side handles business logic and validation, while the query side provides optimized read views.

        Write Side: Validates commands → Produces domain events → Stores in event store or database

        Read Side: Subscribes to events → Projects into read-optimized views (denormalized, pre-aggregated) → Serves queries

        CQRS allows independent scaling (often reads >> writes), different data models optimized for their purpose, and pairs naturally with Event Sourcing.

        💻 CQRS read model — Event-driven projection for query optimization
        // projections/order-summary.projection.ts — Build denormalized read model from events import { Kafka } from 'kafkajs'; import { PrismaClient } from '@prisma/client'; const prisma = new PrismaClient(); const kafka = new Kafka({ brokers: [process.env.KAFKA_BROKERS] }); const consumer = kafka.consumer({ groupId: 'order-summary-projection' }); // Separate read-optimized schema (denormalized for fast queries) // schema: OrderSummary { id, customerId, customerName, status, // itemCount, totalAmount, lastUpdated } async function startProjection() { await consumer.connect(); await consumer.subscribe({ topics: ['order.events', 'customer.events'] }); await consumer.run({ eachMessage: async ({ topic, message }) => { const event = JSON.parse(message.value.toString()); switch (event.type) { case 'ORDER_PLACED': await prisma.orderSummary.create({ data: { id: event.payload.id, customerId: event.payload.customerId, status: 'PENDING', itemCount: event.payload.items.length, totalAmount: event.payload.total, lastUpdated: new Date(), }, }); break; case 'ORDER_SHIPPED': case 'ORDER_DELIVERED': case 'ORDER_CANCELLED': await prisma.orderSummary.update({ where: { id: event.payload.orderId }, data: { status: event.type.replace('ORDER_', ''), lastUpdated: new Date() }, }); break; case 'CUSTOMER_UPDATED': // Denormalized: update customer name across all their orders await prisma.orderSummary.updateMany({ where: { customerId: event.payload.id }, data: { customerName: event.payload.name }, }); break; } }, }); } // Query the read model — fast, no joins, no N+1 // GET /api/orders/summary?customerId=123&status=SHIPPED app.get('/api/orders/summary', async (req, res) => { const summaries = await prisma.orderSummary.findMany({ where: { customerId: req.query.customerId, status: req.query.status }, orderBy: { lastUpdated: 'desc' }, take: 50, }); res.json(summaries); });
        Saga Pattern for Distributed Transactions

        A saga is a sequence of local transactions in different services. Each step either succeeds and triggers the next step, or fails and triggers compensating transactions to undo previous steps.

        Example — Order Processing Saga:

        1. Create Order (Order Service) → 2. Reserve Items (Inventory Service) → 3. Process Payment (Payment Service) → 4. Ship Order (Shipping Service)

        If step 3 fails: Compensate step 2 (Release inventory) → Compensate step 1 (Cancel order)

        Choreography: Each service emits events, no central coordinator. Pros: Simple, loosely coupled. Cons: Difficult to track, cyclic dependencies possible.

        Orchestration: A saga orchestrator coordinates steps. Pros: Clear flow, easy to understand. Cons: Central point of control, must avoid business logic leaking into orchestrator.

        Change Data Capture (CDC)

        CDC captures row-level changes in a database and publishes them as events. Debezium is the most popular open-source CDC platform. It reads the database's transaction log (WAL in PostgreSQL, binlog in MySQL) and produces change events to Kafka.

        Use cases: Maintaining read replicas, populating search indices, synchronizing data between services, implementing the Outbox pattern. CDC ensures reliable event publishing without dual-write problems.

        💻 Transactional outbox — atomic write + event publish

        Caching Strategies

        Caching reduces database load and latency by storing frequently accessed data in fast, in-memory stores like Redis or Memcached. Common patterns include cache-aside (lazy loading), write-through, and write-behind.

        Cache-aside: Application checks cache first; on miss, fetches from DB and populates cache. Write-through: Writes go to cache and DB simultaneously. Write-behind: Writes go to cache first, then async flush to DB. Cache invalidation on writes prevents stale data.

        Deployment Strategies

        Strategy Blue-Green Deployment

        Run two identical production environments (Blue and Green). Deploy new version to the idle environment, run smoke tests, then switch traffic. Instant rollback by switching back. Requires double the infrastructure during deployment.

        Strategy Canary Deployment

        Gradually route traffic to the new version: 1% → 5% → 25% → 100%. Monitor error rates and performance at each stage. Automatic rollback if anomalies are detected. Minimizes blast radius of failed deployments.

        Strategy Rolling Update

        Gradually replace old instances with new ones. Kubernetes default strategy. Configure maxSurge and maxUnavailable to control the update pace. Zero downtime but both versions run simultaneously during the update.

        Strategy Feature Flags

import { Injectable, Inject, NotFoundException } from '@nestjs/common';
import { ClientKafka } from '@nestjs/microservices';
import { PrismaService } from '../prisma/prisma.service';
import { CreateOrderDto } from './dto/create-order.dto';

@Injectable()
export class OrdersService {
  constructor(
    private prisma: PrismaService,
    @Inject('INVENTORY_SERVICE') private kafka: ClientKafka,
  ) {}

  async create(dto: CreateOrderDto) {
    return this.prisma.$transaction(async (tx) => {
      const order = await tx.order.create({
        data: {
          customerId: dto.customerId,
          items: { create: dto.items },
          status: 'PENDING',
        },
        include: { items: true },
      });

      // Transactional outbox — publish after commit
      await tx.outboxEvent.create({
        data: {
          aggregateType: 'Order',
          aggregateId: order.id,
          eventType: 'ORDER_PLACED',
          payload: JSON.stringify(order),
        },
      });

      return order;
    });
  }

  async findOne(id: string) {
    const order = await this.prisma.order.findUnique({
      where: { id },
      include: { items: true },
    });
    if (!order) throw new NotFoundException(`Order ${id} not found`);
    return order;
  }
}

        // src/lib/payment-client.js — Circuit breaker wrapping payment service const CircuitBreaker = require('opossum'); const { logger } = require('./logger'); const PAYMENT_URL = process.env.PAYMENT_SERVICE_URL || 'http://payment-svc:3001'; async function chargePayment(orderId, amount) { const res = await fetch(`${PAYMENT_URL}/charge`, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ orderId, amount }), signal: AbortSignal.timeout(5000) }); if (!res.ok) throw new Error(`Payment failed: ${res.status}`); return res.json(); } const breaker = new CircuitBreaker(chargePayment, { timeout: 5000, errorThresholdPercentage: 50, resetTimeout: 30000, volumeThreshold: 5 }); breaker.on('open', () => logger.warn('Payment circuit OPEN')); breaker.on('halfOpen', () => logger.info('Payment circuit HALF-OPEN')); breaker.on('close', () => logger.info('Payment circuit CLOSED')); breaker.fallback((orderId) => ({ status: 'deferred', message: 'Payment queued for retry', orderId })); module.exports = { callPaymentService: (orderId, amount) => breaker.fire(orderId, amount) };
